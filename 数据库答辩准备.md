## 二、约束（CONSTRAINT）— 数据库的"守门员"

### 答辩怎么说

> "我在建表时使用了多种数据库约束来保证数据完整性：  
> `PRIMARY KEY` 确保每行唯一，`FOREIGN KEY` 确保引用关系正确，`NOT NULL` 保证关键字段不空，  
> `CHECK` 限制字段取值范围，`UNIQUE` 防止学院名和课程名重复。"

### 2.1 你的项目中用到的所有约束类型

#### ① PRIMARY KEY（主键约束）—— "身份证"
```sql
f_college_id VARCHAR(2) PRIMARY KEY,  -- 学院编号，不能重复，不能为空
```

#### ② COMPOSITE PRIMARY KEY（联合主键）—— "组合身份证"
```sql
-- 授课表：同一个教师不能重复教同一门课
PRIMARY KEY (f_course_id, f_teach_id),

-- 选课表：同一个学生不能重复选同一门课+同一个教师
PRIMARY KEY (f_course_id, f_teach_id, f_stu_id),
```

#### ③ FOREIGN KEY（外键约束）—— "关系的锁链"
```sql
-- 学生表的学院编号必须存在于学院表中
FOREIGN KEY (f_college_id) REFERENCES t_college(f_college_id),

-- 课程表的先修课程必须是一门存在的课程（自引用外键！）
FOREIGN KEY (f_pre_course) REFERENCES t_course(f_course_id),
```

#### ④ CHECK（检查约束）—— "安检门"
```sql
f_sex VARCHAR(1) NOT NULL CHECK (f_sex IN ('男', '女')),   -- 性别只能是男或女
f_title VARCHAR(50) CHECK (f_title IN ('教授','副教授','讲师','助教','高工')), -- 职称只能5选1
f_credit NUMERIC(2,1) CHECK (f_credit >= 0 AND f_credit <= 10), -- 学分0-10
```

#### ⑤ UNIQUE（唯一约束）
```sql
f_name VARCHAR(50) NOT NULL UNIQUE,  -- 学院名不能重复
```

#### ⑥ NOT NULL + DEFAULT
```sql
f_password VARCHAR(50) NOT NULL DEFAULT '123456',  -- 初始密码
```

## 三、索引（INDEX）—— 查询加速器

### 3.1 你为什么建索引

```sql
CREATE INDEX idx_stu_course_course_id ON t_stu_course(f_course_id);
CREATE INDEX idx_stu_course_teach_id  ON t_stu_course(f_teach_id);
CREATE INDEX idx_stu_course_stu_id    ON t_stu_course(f_stu_id);
```

### 答辩怎么说

> "选课表是最频繁被 JOIN 查询的表，我在它的三个外键上各建了一个索引。  
> 这就像给书加了目录——没有索引时，数据库要**全表扫描**，一行一行对比；  
> 加了索引后，数据库直接定位到相关行，JOIN 查询速度大幅提升。  
> 尤其是学生查成绩单时，`WHERE f_stu_id = 'xxx'` 有了索引几乎瞬间返回。"

### 3.2 索引的代价（追问时体现思考深度）

> "但索引也不是越多越好——每建一个索引，INSERT/UPDATE/DELETE 时就要多维护一个索引结构。  
> 所以我只在**高频查询字段**上建索引，主键自带索引所以不需要额外建。"

## 六、存储过程（Stored Procedure）

### 6.1 你唯一的存储过程：sp_cancel_course

```sql
PROCEDURE sp_cancel_course(
    IN  p_teach_id   VARCHAR(6),    -- 输入：教师工号
    IN  p_course_id  INT,            -- 输入：课程编号
    OUT p_result     INT             -- 输出：结果码
)
```

### 6.2 执行流程

```
输入 → 检查是否有成绩 → 有 → 返回 1（已考试，不能停开）
                      ↓ 无
                   检查人数 ≥ 20 → 是 → 返回 2（人数≥20，不能停开）
                      ↓ 否
                   删除选课记录 + 删除授课记录 → 返回 0（成功）
```

### 6.3 答辩怎么说

> "停开课程不是一个简单的 DELETE，需要先**校验**：  
> 1. 已经考过试的课程不能停开（学生白考了）  
> 2. 选课人数 ≥ 20 人的不能停开（影响面太大）  
> 
> 这些逻辑我封装成了存储过程 `sp_cancel_course`，用 OUT 参数返回结果码，  
> `app.js` 里通过 `CALL sp_cancel_course(?, ?, @result)` 调用，然后读取 `@result`。  
> 这种写法比在代码里写一堆 if/else 更清晰，而且数据库执行效率更高。"

### 6.4 存储过程的优点

| | 存储过程 | 代码里写 SQL |
|------|----------|-------------|
| **执行速度** | 预编译，快 | 每次都要解析 |
| **网络开销** | 只传参数 | 传输多条 SQL |
| **逻辑集中** | 数据库层面统一 | 散落在各路由函数里 |
| **安全性** | 可只给 CALL 权限不给 TABLE 权限 | 必须给表权限 |

### 6.5 调用方式（app.js 中）

```javascript
await pool.query('CALL sp_cancel_course(?, ?, @p_result)', [teach_id, course_id]);
const [r] = await pool.query('SELECT @p_result AS p_result');
const code = r[0].p_result;
// code = 0 → 成功, 1 → 已考试, 2 → 人数≥20
```

## 七、触发器（Trigger）—— 自动守卫

### 7.1 你唯一的触发器：trg_check_credit

```sql
TRIGGER trg_check_credit
BEFORE INSERT ON t_stu_course      -- 在插入选课记录之前触发
FOR EACH ROW                       -- 每插入一行都触发
```

### 7.2 执行逻辑

```
学生点击"选课"
  ↓
BEFORE INSERT 触发器触发
  ↓
计算该学生已有总学分 ← SUM(所有已选课程的学分)
  ↓
加上新课程的学分
  ↓
≥ 100？→ 是 → SIGNAL 抛出错误，INSERT 被阻止
        → 否 → 放行，INSERT 正常执行
```

### 7.3 答辩怎么说

> "我写了一个 `BEFORE INSERT` 触发器来**自动检查学分上限**。  
> 不用这个触发器的话，我得在 app.js 里先查学分、再判断、再决定插不插——但这里有个**并发漏洞**：  
> 如果两个请求同时选课，可能都查到学分 98，都以为能选，实际上加起来超了。  
> **触发器在数据库层面原子性地执行**，不会有这个问题。  
> 
> `NEW.f_stu_id` 代表即将插入的那行数据的学生学号，我用它算出总学分，  
> 如果 ≥ 100 就用 `SIGNAL` 抛出自定义错误，拦截插入。"

### 7.4 触发器 vs 代码校验

| | 触发器 | 代码校验 |
|------|--------|----------|
| 并发安全 | ✅ 数据库层面原子操作 | ❌ 可能 race condition |
| 遗漏风险 | ✅ 不管从哪个入口插入都会触发 | ❌ 换个 API 入口可能漏检 |
| 调试难度 | 较难 | 较易 |
| 错误提示 | 数据库错误不太友好 | 可以自定义友好提示 |

## 九、GPA 计算（你的特色功能）

### 9.1 绩点映射表

| 分数 | 绩点 | 等级 |
|------|------|------|
| ≥ 90 | 4.0 | 优秀 |
| 85-89 | 3.7 | 良好 |
| 82-84 | 3.3 | 良好 |
| 78-81 | 3.0 | 中等 |
| 75-77 | 2.7 | 中等 |
| 72-74 | 2.3 | 中等 |
| 68-71 | 2.0 | 及格 |
| 64-67 | 1.5 | 及格 |
| 60-63 | 1.0 | 及格 |
| < 60 | 0 | 不及格 |

### 9.2 加权计算公式

```
GPA = Σ(每门课的学分 × 该课绩点) ÷ Σ(已出成绩课程的总学分)

举例：
C语言    4.0 学分 × 4.0 绩点 = 16.0
数据库   3.5 学分 × 2.3 绩点 =  8.05
计算机网络 3.0 学分 × 4.0 绩点 = 12.0
─────────────────────────────────────
合计     10.5 学分    36.05 点
GPA = 36.05 / 10.5 = 3.43
```

### 9.3 你代码里的实现（app.js my-grades 路由）

```javascript
let totalCredit = 0, totalPoints = 0;
graded.forEach(r => {
    const credit = parseFloat(r.f_credit) || 0;
    const score = parseFloat(r.f_score) || 0;
    let gp = 0;
    if (score >= 90) gp = 4.0;
    else if (score >= 85) gp = 3.7;
    // ... 依此类推
    totalCredit += credit;
    totalPoints += credit * gp;
});
const gpa = totalCredit > 0 ? (totalPoints / totalCredit).toFixed(2) : null;
```

## 十、参数化查询与 SQL 注入（安全问题）

### 10.1 什么是 SQL 注入


```javascript
// ✅ 安全写法（你的项目全部如此）
```

## 十一、事务（ACID）

### 11.1 存储过程中的隐式事务

你的 `sp_cancel_course` 存储过程中有**两条 DELETE 语句**：

```sql
DELETE FROM t_stu_course WHERE ...;   -- 先删选课记录
DELETE FROM t_teach_course WHERE ...;  -- 再删授课记录
```

如果第一条成功、第二条失败，就会出现"课停开了但学生选课记录还在"的数据不一致。

**MySQL 存储过程默认每条语句自动提交。** 如果要保证原子性：

```sql
-- 在存储过程中加事务控制
START TRANSACTION;
DELETE FROM t_stu_course WHERE ...;
DELETE FROM t_teach_course WHERE ...;
COMMIT;  -- 两条都成功才提交，否则回滚
```

### 答辩怎么说（如果被问到）

> "目前我的存储过程没有显式事务包装，这是一个可以改进的地方。  
> 如果停开课程操作中任意一步失败，可以通过 `START TRANSACTION ... COMMIT` 包装，  

> 利用 MySQL 的原子性保证——要么两条 DELETE 都执行，要么都不执行。"

## 十二、答辩常见问题速答

### Q1: "为什么用 MySQL 而不是 MongoDB 或 PostgreSQL？"
> "选课系统是典型的**关系型数据**——学生、课程、教师之间有严格的外键关联，MySQL 是经典的关系型数据库，适合本项目场景。MongoDB 是非关系型的，不适合强关联场景。"

### Q2: "什么是三大范式？你的表满足第几范式？"
> "**1NF**：每个字段不可再分（满足）。**2NF**：非主键字段完全依赖于主键（满足，比如选课表的学生姓名不在选课表里，而在学生表里）。**3NF**：非主键字段不传递依赖（满足，所有表没有冗余字段）。"

### Q3: "你的 t_stu_course 表为什么不存学生姓名？"
> "这是为了遵循**第三范式**——消除传递依赖。学生姓名由 f_stu_id 决定，而 f_stu_id 已是主键。如果把姓名也存在选课表，当学生改名时就要同时更新 student 表和 stu_course 表，容易**数据不一致**。"

### Q4: "CHAR 和 VARCHAR 的区别？"
> "`CHAR(6)` 固定占 6 个字符空间，短了补空格。`VARCHAR(6)` 最多 6 字符，实际存多少占多少。我的教师工号用 VARCHAR(6) 是因为工号长度可能不固定。"

### Q5: "LEFT JOIN 和 INNER JOIN 的区别？"
> "INNER JOIN 只返回两表都匹配的行。LEFT JOIN 保留左表全部，右表没匹配的填 NULL。比如统计各学院学生人数时，有的学院可能还没学生，用 LEFT JOIN 能正确显示 0 人而不是直接不显示该学院。"

### Q6: "DELIMITER 是什么意思？"
> "`DELIMITER //` 告诉 MySQL 暂时把 `//` 当成语句结束符，因为存储过程和触发器的定义体内包含 `;` 分号。定义完后 `DELIMITER ;` 恢复分号为结束符。"

### Q7: "为什么用 NUMERIC(2,1) 而不是 FLOAT？"
> "学分和成绩需要**精确**的小数，浮点数有精度问题（比如 0.1+0.2 ≠ 0.3）。`NUMERIC(2,1)` 精确存储一位小数，不会出现浮点误差。"

### Q8: "你的成绩表里 f_score 可以是 NULL，为什么不设 NOT NULL？"
> "学生选课时成绩未知，只有考试后才有成绩。NULL 恰好表示'尚未出分'这个状态。注意这跟 0 分不同——0 分是一个确定的分数。"
